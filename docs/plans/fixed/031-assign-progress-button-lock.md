# 031-assign-progress-button-lock

## Context

振り分け処理は VC 準備や招待リンク生成を含むため、
ボタン押下から結果表示まで数秒待つことがあります。

現状はインタラクションを defer しているものの、
募集 Embed 側には「処理中」が出ないため、
押下が通ったのか分かりづらく、重複操作の不安も残ります。

利用者要望として、
振り分けボタンを `🤖 計算中` に変更して無効化し、
結果表示後に `🎲 振り分け` または `🔁 再振り分け` に戻したいという案が出ています。

この改善は UI 変更だけでは不十分で、
同時押しによる二重実行を防ぐ内部ガードもセットで必要です。

## 変更ファイル

| ファイル | 対応内容 |
|---------|---------|
| `internal/bot/bot.go` | 振り分け中のボタン表示、無効化、完了後の復元、内部ガードを追加 |
| `internal/bot/bot_test.go` | 振り分け中表示、復元、同時押し防止、再振り分け時の回帰テストを追加 |
| `internal/model/recruitment.go` | `AssignInProgress` 状態を追加 |

## 実施順序

1. `Recruitment` に `AssignInProgress` を追加する
2. 先にボタン状態遷移のテストを書く
3. 押下直後に募集 Embed を `計算中` 表示へ更新する
4. 同時押し防止の内部ガードを入れる
5. 結果送信後にボタン表示を復元する

## 要件

### 要件1: 押下直後に募集 Embed を処理中表示へ切り替える

- 発案者が振り分けボタンを押したら、募集 Embed の振り分けボタンを即時更新する
- ラベルは `🤖 計算中` とする
- 振り分けボタンは無効化する

#### 期待結果

- 利用者が「押下は受理された」と分かる
- 再クリックしようとしたときに UI 上でも抑止される

### 要件2: 結果表示後に適切なラベルへ戻す

- 初回振り分けの完了後は `🔁 再振り分け` にする
- 再振り分けでも同様に、押下中は `🤖 計算中`、完了後は `🔁 再振り分け` に戻す
- 振り分け失敗時も、利用者が詰まらない形でボタン状態を復元する

### 要件3: 内部ガードで二重実行を防ぐ

- UI 上の無効化だけに依存しない
- 同じ募集に対する `assign` が処理中なら、後続の `assign` は拒否する
- 競合時はエフェメラルで「現在振り分け中です」を返す

#### 確認したいケース

- 連打や複数端末操作でも振り分けが二重実行されない
- VC 作成や招待リンク生成が重複しない

### 要件4: 既存の権限制御と人数条件を壊さない

- 発案者のみ実行できる制約を維持する
- 10人未満では引き続き振り分け不可とする
- 募集終了後の操作拒否も維持する

### 要件5: テストで状態遷移を固定する

- 押下直後に `🤖 計算中` へ変わること
- 完了後に `🔁 再振り分け` に戻ること
- 同時押しが拒否されること
- 失敗時にボタンが復元されること
- `AssignInProgress` が開始時に `true`、終了時に `false` へ戻ること

## 検証方法

- `go test ./internal/bot/...`
- 実機で振り分け時にボタンが `🤖 計算中` へ変わることを確認
- 振り分け結果送信後に `🔁 再振り分け` として再度押せることを確認
- 連打時に二重投稿や VC 重複作成が起きないことを確認

## 結果

- 2026-03-01: 完了
- `internal/model.Recruitment` に `AssignInProgress` を追加し、振り分け開始時の `🤖 計算中` 表示、完了後の `🎲 振り分け` / `🔁 再振り分け` 復元、失敗時の復元を実装済み
- `internal/bot/bot.go` に同一募集の二重実行を拒否する内部ガードを追加し、競合時は `現在振り分け中です` のエフェメラル応答を返すよう更新済み
- `internal/bot/bot_test.go` に処理中表示、復元、競合拒否、成功経路の回帰テストを追加し、`go test ./internal/bot/...` で確認済み
