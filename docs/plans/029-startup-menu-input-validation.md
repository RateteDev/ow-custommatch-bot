# 029-startup-menu-input-validation

## Context

起動メニューは 022 と 026 で改善されましたが、
現在の入力処理は `1` 以外の想定外入力でも通常運用として続行されます。

現状は短いタイムアウトと単発入力の組み合わせになっており、
利用者が数字を押し損ねた場合や対象外の数字を入力した場合でも、
そのまま本番起動してしまう余地があります。

誤操作で通常運用に入る挙動をなくし、
利用者が明示的に起動方法を選べるメニューへ見直します。

## 変更ファイル

| ファイル | 対応内容 |
|---------|---------|
| `cmd/ow-custommatch-bot/main.go` | 起動メニューの入力待機・再入力制御を改善 |
| `cmd/ow-custommatch-bot/main_test.go` | 有効入力・無効入力・Enter・タイムアウトのテストを追加/更新 |
| `docs/CLI表示仕様.md` | 起動メニューの最終仕様を更新 |

## 実施順序

1. 現在の入力仕様を整理する
2. Enter / 有効数字 / 無効数字の挙動を固定する
3. 先にテストを追加する
4. 入力ループを実装する
5. 仕様書を更新する

## 要件

### 要件1: 有効な入力だけを受け付ける

- 起動メニューでは `1`、`2`、`3`、空入力のみを受け付ける
- 空入力は通常運用として扱う
- `4` など対象外の数字、文字列、記号はエラー表示して再入力させる

#### 完了条件

- 想定外入力で通常運用が始まらない
- 利用者が再入力の必要性を画面上で理解できる

### 要件2: 数字入力を待機できるようにする

- 起動メニューはタイムアウトによる自動起動を廃止する
- 対話的な起動では、利用者が `1`、`2`、`3`、`Enter` のいずれかを入力するまで待機する
- 無効入力後も同じメニューを再表示し、正しい入力があるまで待機する

### 要件3: Enter で通常運用を維持する

- 何も入力せず `Enter` を押した場合は通常運用を開始できる
- 従来の「すぐ始めたい人」の導線は残す

### 要件4: テストで誤起動を防ぐ

- `1` で通常運用になること
- `2` で動作確認用になること
- `3` でトークン上書きに進むこと
- 空入力で通常運用になること
- 無効入力後に再入力できること
- 自動起動しないことをテストで固定すること

## 検証方法

- `go test ./cmd/ow-custommatch-bot/...`
- 実行ファイル起動時に `4` や `abc` を入力し、再入力を求められることを手動確認
- `Enter` のみで通常運用を開始できることを手動確認
