# 028-makefile-help-comments

## Context

`Makefile` の各ターゲットは意味を把握していれば使えますが、
利用者コメントとして「それぞれのコマンドに日本語の説明コメントが欲しい」という要望が出ています。

現状は、変数やターゲット名から意味を推測する必要があり、
初めて触る人にとっては `build`、`build-win`、`release-win-exe` の違いが直感的ではありません。

開発者が `Makefile` を開いたときに目的をすぐ理解できるよう、
各ターゲットと主要変数の役割を短い日本語コメントで明示します。

## 変更ファイル

| ファイル | 対応内容 |
|---------|---------|
| `Makefile` | 各変数とターゲットに日本語説明コメントを追加 |
| `README.md` | 必要であれば代表的なターゲットの説明を補足 |

## 実施順序

1. 各変数とターゲットの用途を棚卸しする
2. `Makefile` に短い日本語コメントを追加する
3. 必要なら README の開発者向け説明を補う

## 要件

### 要件1: 各ターゲットに目的説明を付ける

- `test`、`build`、`run`、`build-win`、`release-win-exe` の目的が分かるコメントを付ける
- コメントは「何をするターゲットか」が一読で分かる短さにする

#### コメント例

```make
# テストを実行します
test:
	go test ./...

# Windows 用 exe をビルドします
build-win:
	...
```

### 要件2: 主要変数にも補足を付ける

- `APP_NAME`、`CMD_PATH`、`BIN_DIR`、`DIST_DIR` など、
  用途がすぐ伝わらない主要変数にも短い説明を付ける

#### 完了条件

- `Makefile` を上から読んだとき、変数とターゲットの役割を追いやすい
- コメントが長すぎて本体を読みにくくしない

### 要件3: 保守しやすいコメントにする

- コメントは手順説明ではなく目的説明に寄せる
- 実装変更で古くなりにくい書き方にする

#### 悪い例

- 「まず mkdir してから go build して...」のような詳細手順を書くこと

#### 良い例

- 「開発用のローカルビルドを行う」
- 「Release 用の Windows exe を生成する」

### 要件4: 必要に応じて README を補う

- `Makefile` だけでは伝わりにくい場合は、README に代表的な利用例を 2〜3 個だけ補う
- ただし主役は `Makefile` 内コメントであり、README に説明を分散させすぎない

## 検証方法

- `make test`
- `make build`
- `make release-win-exe`

---
ユーザーコメント：READMEには書かなくていいよ。Makefileのコメントだけで十分。
