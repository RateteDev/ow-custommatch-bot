ow-custommatch-bot の現行ロードマップです。
詳細設計は `docs/plans/` 配下の各 plan を参照してください。

## 現在の前提

- `docs/plans/fixed/001`〜`024` は完了済みまたはクローズ済みです
- 今後の実装対象は `025`〜`028` です
- 実装は **1プランずつ** 進め、各 plan の「実施順序」「検証方法」に従って完了判定します
- 直近のユーザーコメントで、`026` と `027` は起動時 UI の導線として合わせて考える方針が見えています
- `028` は独立度が高いため、他フェーズの合間でも実施できます

## 実施方針

### フェーズ1: `/match` 再実行 UX 改善

#### [025](plans/025-match-restart-confirm.md)

- 同一 channel で `/match` を再実行した際に確認 UI を表示する
- 「閉じて開始」「キャンセル」の分岐と権限制御を追加する
- Bot の回帰テストで再実行時の挙動を固定する

注意点:
- 既存募集と新規募集が同時に open にならないことを最優先に確認する
- 確認 UI 表示後に状態が変わっていても壊れないことを意識する
- 発案者以外の操作や古い UI からの操作を安全側で拒否する

### フェーズ2: 起動 UI の仕上げ

#### [027](plans/027-launch-ui-polish.md)

- バナー枠内にツール名、バージョン、URL を集約する
- `データ保存先` は削除し、必要最小限のパス表示にする
- 接続成功後の案内は、少し派手さを持たせつつ `/match`、ランク登録、確認系コマンドの導線を入れる
- `main_test.go` と `docs/CLI表示仕様.md` を最終表示に合わせて更新する

注意点:
- 起動時の情報量を増やしすぎず、最初に見るべき情報から順に並ぶこと
- コマンド名は現行実装にあるものだけを出すこと
- ANSI 色付けを使う場合は、色なし環境でも読める文面を保つこと

### フェーズ3: TOKEN 運用導線の整備

#### [026](plans/026-token-store-management.md)

- CLI オプション中心ではなく、起動時のモード選択に近い導線で「トークンを上書きする」を扱う方向で進める
- 利用者向けには「どこに保存されるか」「どう上書きするか」を分かるようにする
- 削除方法は利用者向けではなく、[`developer.md`](developer.md) 側に整理する

注意点:
- 一般利用者にトークン削除を露出しすぎないこと
- 起動フローを複雑にしすぎず、通常起動の邪魔をしないこと
- 027 の起動 UI 変更と競合しやすいため、メニュー構造を二重にいじらないこと

### フェーズ4: 開発者向け補足

#### [028](plans/028-makefile-help-comments.md)

- `Makefile` の各ターゲットと主要変数に短い日本語説明コメントを付ける
- README は触らず、`Makefile` 内コメントだけで完結させる
