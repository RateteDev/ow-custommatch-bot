# マッチングアルゴリズム改善案

## 前提（現行ロジックの要点）
- 参加者をスコア降順に並べ、上位・中位・下位の 3 層に分割する。
- ランダム試行を 20 回行い、チーム平均スコア分散が最小の組み合わせを採用する。
- 端数プレイヤーはシャッフル後に余りとして扱う。

この方式は「平均スコアの均等化」には強い一方、ロール偏り・フレックス偏り・連続同チームなどの体験品質を直接最適化していない。

---

## 改善方針（後方互換性を気にしない前提）

### 1. 目的関数を単一指標から多目的スコアへ拡張
現在は平均スコア分散のみを最小化しているため、以下を同時評価する。

- **チームレーティング差**（現行指標）
- ~~**ロール充足度**（Tank/DPS/Support の不足・過多ペナルティ）~~
- ~~**フレックス過多ペナルティ**（全員フレックスで役割不明になる状態を抑制）~~
- **重複ペア抑制**（直近 N 試合で同チームだった組み合わせを減点）
- **待機公平性**（端数落ちが連続するプレイヤーに救済ボーナス）

> 合計コストを `総合コスト = w1*レート差 + w2*ロール不足 + w3*重複ペア + w4*待機不公平` のように定義し、最小化対象を置き換える。

### 2. ランダム探索を「初期解 + 近傍探索」に変更
現行の 20 回ランダム試行は軽量だが局所的な当たり外れが大きい。以下へ変更する。

1. **初期解生成**: スネークドラフト（1→N→N→1）でレートを粗く均等化。
2. **近傍探索**: 2 人交換 / 3 人循環交換でコストが下がる操作を反復。
3. **焼きなまし（任意）**: 一時的な悪化も許容して局所最適を脱出。
4. **打ち切り条件**: 100ms〜300ms または改善停止回数で終了。

これにより、実行時間を制御しつつランダム性依存を下げられる。

### 3. 端数（余り）処理を「公平キュー化」
現行は毎回シャッフルで余りを決めるため、同一プレイヤーが連続で落ちる可能性がある。以下へ変更する。

- プレイヤーごとに `bench_count`（最近何回余ったか）を持つ。
- 余り選定時に `bench_count` が低い人を優先的に余りへ回す。
- マッチ参加した人は `bench_count` を減衰、余りになった人は増加。

これで「体感不公平」を明示的に抑制できる。

### 4. ロール情報を導入した制約付き編成
~~OW の体験品質に直結するため、プレイヤーの希望/得意ロールを保持して制約化する。~~

- ~~**ハード制約**（必須）: 各チーム最低 1 Tank, 2 DPS, 2 Support など。~~
- ~~**ソフト制約**（望ましい）: 第一希望ロール一致に加点。~~
- ~~**不足時フォールバック**: ハード制約を段階的に緩和（例: 2-2-1 を許可）。~~

> コメント: 個人の希望ロール・得意ロールを前提にすると入力/管理コストが高く、運用複雑性が上がりすぎるため、この案はいったん見送る。

### 5. ランクスコアを固定値から不確実性込みへ
~~単一ランク値は直近の調子やサブロール実力を表現できない。以下を検討する。~~

- ~~**区間レート**（平均 ± 不確実性）を持つ。~~
- ~~サブロール時はメインロールより係数を下げる。~~
- ~~試合後に簡易更新（勝敗・KDA ではなく、まずは自己申告評価でも可）。~~

~~将来的に TrueSkill/Elo 互換の更新式へ拡張しやすくなる。~~

> コメント: 個人のロール別ランク更新まで踏み込むとモデル・UI・運用負荷が大きいため、現時点では採用しない。

### 6. 再現性とデバッグ性の確保
改善後はロジックが複雑化するため、運用時トラブル調査が可能な設計にする。

- マッチごとに `seed` を保存して再現可能にする。
- 採用解のコスト内訳（レート差/ロール不足/重複ペア）をログ出力。
- 「なぜこの編成か」を説明できる簡易テキストを表示。

---

## 実装優先順位（推奨）
1. **多目的コスト関数化**（現行構造を保ったまま効果が高い）
2. **端数公平キュー**（ユーザー体験への即効性が高い）
3. **近傍探索導入**（品質を底上げ）
4. ~~**ロール制約対応**（データ入力 UI とセットで実装）~~
5. ~~**レート更新モデル拡張**（中長期施策）~~

---

## 最小実装案（2週間想定）
- Week 1
  - コスト関数を構造体化（重み付き）
  - 端数公平キュー（`bench_count`）を導入
  - seed 固定実行とログ出力を追加
- Week 2
  - 初期解 + 2 人交換の近傍探索を追加
  - 回帰テスト（同 seed で同結果）を整備
  - コスト推移の計測を追加

この最小実装だけでも、現行の「ランダム依存」「余り不公平」「調整理由の不透明さ」を大きく改善できる。
